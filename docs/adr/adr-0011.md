# ADR-0011: Semantic Embedding via llama.cpp + Replaceable GGUF Embedder

## Status
Proposed

## Date
2026-06-09

## Context

NoesisNoema markets itself as a private on-device RAG system. On-device UAT
(2026-06-08, Spinoza Ethica RAGpack, Llama 3.2 3B) demonstrated that retrieval is
**not actually semantic**. Five RAG-verification questions produced plausible but
unverifiable answers; the decisive question requiring verbatim source text returned
correct content only by lucky guess, immediately followed by chat-template residue
spirals. Cross-checking cited proposition numbers against Spinoza's actual text
revealed multiple confabulations.

Root cause: `Shared/RAG/EmbeddingModel.swift` produces 10-dimensional pseudo-hash
vectors from character codes — a fingerprint, not a semantic embedding. Meanwhile
`noesisnoema-pipeline` (the canonical RAGpack producer) emits 384-dimensional
`sentence-transformers/all-MiniLM-L6-v2` vectors in RAGpack v1.1. The app does not
even know how to read the `embeddings.npy` file the pipeline produces; it looks for a
v0.x-shaped `embeddings.csv` instead, finds nothing, and returns silently. The
"RAG-shaped" answers users have been seeing are general-knowledge LLM output dressed
with hallucinated citations — a clean ADR-0000 violation.

This must be fixed before v0.4 ships. The fix introduces two non-negotiable design
constraints, derived from the product owner's stated long-term direction:

1. **The embedding model is replaceable, not fixed.** OSS LLMs and embedding models
   improve on roughly half-yearly cadence. NoesisNoema's stance is to track the
   field — swap in better models as they emerge. Any architecture that hard-codes
   one embedder identity becomes technical debt the moment that identity is no
   longer best-in-class. The previous draft of this ADR (CoreML + MiniLM) committed
   exactly this mistake and is superseded by the present text.
2. **The embedding runtime is the same llama.cpp xcframework already in use.**
   NoesisNoema already vendors a working llama.cpp xcframework (built with metal,
   already used for LLM inference, already exercised on iOS and macOS). llama.cpp
   supports embedding-mode inference (`llama_context_params.embeddings = true` →
   `llama_get_embeddings(ctx)`). Reusing this stack means: no new ML runtime, no
   CoreML conversion step, GGUF as the single artifact format the app understands,
   and "swap the model" reduced to "drop in a new GGUF file."

These two constraints are the central decisions of this ADR. Implementation details
(which GGUF to ship first, exact pooling strategy, normalisation conventions) flow
from them.

### Option survey (recorded for completeness)

- **CoreML conversion of a specific sentence-transformers model.** Considered and
  rejected: hard-codes a model identity, adds a second ML runtime, and contradicts
  constraint 1.
- **Apple `NLEmbedding.sentenceEmbedding`.** Considered and rejected: model identity
  is opaque, not exchangeable, and the pipeline cannot easily produce matching
  vectors. Locks the system to whatever Apple ships.
- **llama.cpp embedding mode with a replaceable GGUF.** Selected. Honours both
  constraints, reuses existing infrastructure, and aligns app-side and pipeline-side
  on the same artifact format.

## Decision

### 1. Embedding inference runs in the existing llama.cpp xcframework

A new code path in the Swift llama.cpp shim opens an embedding-mode `llama_context`
(`embeddings = true`), tokenises input, runs a forward pass, and returns a pooled
float32 vector from `llama_get_embeddings`. Pooling strategy and any L2
normalisation are recorded in the RAGpack manifest (see Decision 5) so that pipeline
and app converge on identical vectors for identical inputs.

No new ML framework is introduced. No Python, no subprocess, no PythonKit, no
CoreML, no NaturalLanguage. The strict pure-Swift posture is preserved.

### 2. The embedder model is a GGUF, treated as a replaceable artifact

The embedding model ships as a GGUF file (alongside, not instead of, the LLM GGUF
already shipped). Like the LLM GGUF and the llama.cpp xcframework, the embedder
GGUF is **vendored out-of-git** (`.gitignore` extended; build script provisions it;
`Bundle.main` resolves it at runtime).

Initial embedder selection (e.g. `nomic-embed-text-v1.5`, `bge-small-en-v1.5`, or
similar; concrete choice determined at implementation time based on size / quality
trade-off) is recorded but not architecturally privileged. Subsequent ADRs may swap
the embedder; this ADR does not need amendment to do so. The selection criterion is
"best on-device quality at acceptable size on Apple Silicon," not loyalty to any
specific lineage.

### 3. The embedder is identified by content, not by name alone

Because the embedder is intended to evolve, the app and pipeline must be able to
detect when a RAGpack's embeddings were produced by a model the app no longer
agrees with. The RAGpack manifest carries an **embedder identity fingerprint** —
minimally: GGUF SHA-256 (or an equivalent stable hash), embedding dimension,
pooling strategy, normalisation flag, and a human-readable model name. At import,
the app verifies the manifest's fingerprint matches its current embedder. If it does
not, the import is rejected with a clear, user-readable error (see Decision 4).

This shifts identity from "MiniLM forever" to "whichever GGUF this pack was built
with, verified at import." The app may at any time ship a different embedder; old
packs are explicitly known to be incompatible rather than silently giving wrong
retrieval results.

### 4. No silent fallback (ADR-0000 §4)

The current import implementation hides failures behind `print` + `return`. That
ends. The new reader throws a structured error on:

- missing required files in the pack (`chunks.json`, `embeddings.npy`,
  `manifest.json`);
- manifest schema violation;
- embedder fingerprint mismatch between manifest and current app embedder;
- `embeddings.npy` parse failure (bad magic, truncation, wrong dtype, wrong shape);
- length mismatch between chunks and embeddings;
- dimension mismatch between embeddings and the app's runtime query vectors.

The UI surfaces these via alert (matching the iOS/macOS pattern PR #93 established
for inference errors). The user can then either (a) regenerate the pack with a
compatible pipeline, or (b) explicitly switch the app to a different embedder if
one is available.

### 5. Pipeline alignment: RAGpack v1.2

`noesisnoema-pipeline` is updated alongside this ADR to support a v1.2 manifest
shape that records the embedder identity fingerprint described in Decision 3. The
pipeline switches its embedder from `sentence-transformers/all-MiniLM-L6-v2` to the
same GGUF the app ships, run via `llama-cpp-python` (or equivalent llama.cpp
binding), so that **identical input text produces identical output vectors on both
sides**.

Pipeline-side changes are not in this repository, but are explicitly in scope of
the v0.4 release and must be coordinated. A separate ADR or migration note in the
pipeline repo may record the v1.2 spec; this ADR records only that the app
side requires it.

### 6. Existing v1.1 packs are a v0.4 break

RAGpacks already produced by the pipeline under v1.1 (MiniLM-based) become
incompatible. They will be rejected at import with a clear error indicating the
embedder fingerprint mismatch. Users (currently: Taka) regenerate any needed packs
once with the v1.2 pipeline.

This break is accepted as part of v0.4. The alternative — supporting both
embedders in parallel — would compromise constraint 1 (replaceability) by
permanently tying the app to MiniLM. It also has limited practical value: the only
existing packs are internal development artifacts.

### 7. Scope boundaries

This ADR does NOT:

- bundle a specific LLM (Llama 3.2 3B remains the current LLM by separate decision);
- amend ADR-0008 (execution path), ADR-0009 (session memory), or ADR-0010 (macOS UI);
- change the chat template (template residue / `|>` artefacts are a separate
  parked task);
- bundle additional GGUF LLMs;
- introduce remote inference of any kind.

It DOES introduce:

- a llama.cpp embedding-mode wrapper in the Swift shim;
- a new `EmbeddingModel` implementation backed by the above (replacing the current
  hash-based stub);
- a RAGpack v1.2 reader (`.npy` parser + manifest validator + chunks/citations
  loader);
- `.gitignore` entries for the new embedder GGUF;
- error surfaces in the UI for import failures;
- a documented build-time step to provision the embedder GGUF (mirroring the
  existing xcframework and LLM GGUF procedures).

## Rationale

- **Replaceability is the product's long-term posture.** Locking to one embedder
  identity to "make packs portable" trades a permanent constraint for a short-term
  convenience. The cost of being locked in compounds; the cost of regenerating a
  pack is paid once.
- **One runtime is better than two.** Reusing llama.cpp for embeddings means one
  binary stack, one artifact format (GGUF), one set of build tooling, one mental
  model. Introducing CoreML (or any second runtime) for embeddings alone doubles
  these axes without doubling capability.
- **Identity by fingerprint, not by name.** Recording the actual GGUF hash + dim +
  pooling in the manifest lets the system safely evolve. Recording only a model
  name (e.g. `"all-MiniLM-L6-v2"`) is fragile: two different conversions or two
  different quantisations of the "same" model are not actually the same in
  embedding space.
- **Visible failure beats invisible corruption.** Silent import return is the
  worst kind of bug — the user believes retrieval is working when it isn't.
  Throwing on every mismatch case is verbose but correct (ADR-0000 §4).

## Consequences

### Positive
- NoesisNoema performs real on-device semantic retrieval, using the same runtime
  it already uses for LLM inference.
- The embedder can be swapped to a better OSS model whenever one emerges, without
  re-architecting.
- The app and pipeline share a single artifact format (GGUF) and a single
  fingerprint identity, removing an entire class of "which model was this pack
  built with?" ambiguity.
- ADR-0000 §4 restored at the import boundary.
- The hallucination failure mode (Anna Rehlinger; misattributed propositions) loses
  its main enabler: retrieval that returns nothing useful.

### Trade-offs
- Existing v1.1 packs (MiniLM-based) stop working. Accepted as v0.4 break.
- The build/packaging procedure gains one more provisioned artifact (the embedder
  GGUF). Documented as part of the build procedure.
- llama.cpp embedding mode has slightly less Apple-Silicon-tuned performance than
  CoreML for some models; mitigated by the existing metal backend in the
  vendored xcframework, and by choosing a small embedder model (e.g. <500MB GGUF).
- Pipeline-side work is required and must be coordinated; recorded explicitly in
  Decision 5.

### Mitigations
- The manifest fingerprint makes pack incompatibility an explicit, recoverable
  error rather than silent corruption.
- Build script provisioning mirrors the proven xcframework / LLM GGUF flow — no
  new operational pattern.
- llama.cpp embedding mode is well-trodden upstream; not exotic.

## Compliance

Subordinate to ADR-0000 (Human Sovereignty; no silent fallback; no hidden
behaviour). Honours the strict pure-Swift / no-Python / llama.cpp-via-xcframework
posture recorded in the project's standing constraints. ADR-0005 (client routing),
ADR-0008 (execution path), ADR-0009 (session memory), ADR-0010 (macOS UI shell) are
not amended; only the data feeding them changes. ADR-0006 (Contract Lock) is not
broken: `Chunk` and `NoemaRequest` are unaffected; manifest schema changes are on
the pipeline ↔ pack boundary, not the app's public Swift contracts. Reinforces
the principle that NoesisNoema treats OSS models as a moving target it tracks,
not a fixed dependency it inherits.
