# ADR-0009: Session Memory — Multi-Turn Conversation Context

## Status
Proposed

## Date
2026-05-26

## Context

NoesisNoema v0.4 processes every query **statelessly**. `NoemaRequest` carries only
`query` + `sessionId`; `LocalExecutor` builds the generation context from retrieved
knowledge chunks alone. No prior conversation turn is ever passed to the model.

On-device UAT (2026-05-25, Llama 3.2 3B) exposed the consequence: a follow-up like
"Continue describing the war" produces "This conversation just started, I haven't provided
any answers yet." The model has no memory of the immediately preceding turn, so multi-turn
dialogue is impossible — a severe UX defect for a chat product.

The RAGfish reference design already anticipated this. `design/execution-flow.md`:
- Step 5A: "Session memory injected (if within 45 min window)"
- Section 4 (Memory Handling): Scope = client-scoped; Duration = 45 minutes; automatic
  purge on timeout; "No persistent memory unless explicitly approved by user."

So session memory is **specified but unimplemented** in the v0.4 path. This ADR fixes the
contract for implementing it.

There is a hard constraint from the constitution. ADR-0000 §4 ("Memory Injection Without
Disclosure") states: **"No hidden conversation memory."** Any memory mechanism that feeds
the model conversation content the user cannot see would violate the product constitution.
This ADR must therefore define session memory so that it is transparent by construction.

## Decision

### 1. Session memory = the visible conversation, nothing more

Session memory consists **only** of recent Q&A turns that are already visible to the user
in the chat transcript. NoesisNoema MUST NOT inject any conversation content the user
cannot see (ADR-0000 §4). There is no hidden or background memory store feeding the model.

### 2. Bounds: recent 3 turns AND within a 45-minute window

When building the prompt for a local query, the executor includes prior turns subject to
**both** caps:
- **Recency window:** only turns within the last 45 minutes (per execution-flow.md §4).
- **Turn cap:** at most the **3 most recent** Q&A turns (on-device latency / context-window
  budget for a 3B model; retrieved chunks + history + system prompt must coexist).

If either bound excludes a turn, it is omitted. Empty history ⇒ behaves exactly as today
(single-turn).

### 3. Client-scoped, ephemeral, no new persistence

Session memory is sourced from the existing client-side transcript
(`DocumentManager.qaHistory`) — the same data the History tab shows. This ADR introduces
**no new persistent store** and no server session object (NoesisNoema is offline-first;
there is no server). "Persistent memory" beyond the existing, user-visible, user-clearable
transcript is out of scope and would require explicit user approval (execution-flow.md §4).

### 4. Memory does not alter routing, privacy, or retrieval

Session memory is a **prompt-construction** concern only. It MUST NOT change:
- routing decisions (Router/PolicyEngine stay deterministic on the current query),
- the privacy guarantee (history is local data; the local path remains network-incapable,
  ADR-0008 R3 / Step 4.5),
- retrieval (chunks are still retrieved for the current query).

History is concatenated into the prompt alongside retrieved context; it is not a retrieval
input and not a routing signal.

### 5. Carried explicitly through the contract

History travels as an explicit field on `NoemaRequest` (e.g. `history: [QAPair]` or an
equivalent value type), populated by the UI from the visible transcript. The executor does
not reach into global mutable state to discover history (consistent with ADR-0008's removal
of side-effect state coupling). This keeps the data flow inspectable.

## Rationale

- **Transparency by construction (ADR-0000 §4).** By defining session memory as exactly
  the turns the user can already see, "no hidden conversation memory" holds trivially —
  there is nothing hidden to inject.
- **Why both a time window and a turn cap.** The 45-minute window honours the existing
  spec and matches user intuition ("the conversation I'm having now"). The 3-turn cap is an
  engineering bound: a 3B model on-device has a limited context budget shared with retrieved
  chunks; unbounded history would crowd out retrieval and inflate latency.
- **Why no new persistence.** The transcript already persists client-side and is
  user-clearable (the Clear History action, PR #89). Reusing it avoids a second source of
  truth and keeps the "no persistent memory unless approved" rule intact.
- **Why prompt-only, not retrieval/routing.** Keeping memory out of routing preserves
  deterministic routing (ADR-0005) and the privacy enforcement (ADR-0008 R3). Memory
  changes what the model *reads*, not what path the query *takes*.

## Consequences

### Positive
- Multi-turn dialogue works; follow-up questions retain context.
- The privacy and determinism guarantees are untouched.
- No new storage, no new persistence surface, no constitutional conflict.

### Trade-offs
- Including history grows the prompt, costing some latency and context budget on-device —
  bounded by the 3-turn / 45-min caps.
- Very long single answers in history could still pressure the context window; a future
  refinement may summarise or truncate individual turns (out of scope here).

### Mitigations
- Hard caps (3 turns, 45 min). Empty/expired history degrades gracefully to single-turn.
- Prompt formatting reuses the existing chat template; no template change in this ADR (the
  Llama 3.2 vs ChatML template question is tracked separately and must not be conflated with
  this work).

## Compliance

Subordinate to ADR-0000 (esp. §4 "No hidden conversation memory"). Implements
`design/execution-flow.md` Step 5A and §4 (Memory Handling). Does not amend ADR-0005
(routing determinism), ADR-0006 (contract lock — `NoemaRequest` gains an additive,
defaulted field; existing call sites remain source-compatible), or ADR-0008 (execution
layer / privacy enforcement). No silent behaviour: empty history ⇒ identical to current
single-turn behaviour.
