# ADR-0008: v0.4 Layer Migration Completion & Coordinator Unification

## Status
Proposed

## Date
2026-05-21

## Context

NoesisNoema v0.3 shipped as a **monolith**. `ModelManager.generateAsyncAnswer`
performs knowledge retrieval, local llama.cpp inference, and answer assembly in a
single method, under an implicit "always local" assumption and with no routing
concept.

v0.4 — the Noema Architecture work tracked through EPIC1 and EPIC2 — introduced a
correct decision/execution/knowledge separation aligned with ADR-0004 and ADR-0005:
a deterministic, client-side `Router`; a `PolicyEngine`; a `ConstraintRuntime`; and
an executor layer (`LocalExecutor` / `AgentExecutor`) behind a coordinator.

A code audit on 2026-05-21 (NoesisNoema `main`, HEAD `d634006`) established that
**the migration was started but never completed**, leaving the codebase in a
non-compliant intermediate state:

1. **The execution layer's local path is hollow.** `LocalExecutor.execute`
   returns the literal `"[LOCAL LLM STUB] \(query)"`. All real local inference
   still exists only in the v0.3 monolith (`ModelManager`), beside the new
   runtime rather than inside it.

2. **The v0.3 UI bypasses routing entirely.** The shipping screens (`ChatView`,
   `MobileHomeView`, `HistoryView`, `SettingsView`) call `ModelManager` directly.
   On those screens, the on/offline routing that is the entire purpose of v0.4
   does not execute. The only screen that routes (Minimal Client Interface)
   reaches the stub.

3. **Privacy enforcement is decided but not enforced.** The Router correctly
   decides a local route, but no component guarantees zero network egress on a
   local-only request, as required by the Router Decision Matrix and
   `design/execution-flow.md` Step 4.5. `LocalExecutor` has no network awareness.

4. **Duplicate constructs exist**, evidencing an abandoned mid-migration:
   - `Router` vs `HybridRouter` — the latter self-documents "no longer used in the execution path"
   - `PolicyEngine` vs `HybridPolicyEngine` — the latter self-documents the same
   - `ExecutionCoordinator` (referenced only in a SwiftUI Preview) vs
     `HybridExecutionCoordinator` (wired to `@main`)

The product intent for v0.4, reaffirmed 2026-05-21, is **default-offline private
inference with human-sovereign routing to an online path only when the human
chooses**. RAG is the knowledge-supply mechanism, not the product's subject —
consistent with ADR-0000 ("AI is not the subject"). The residual v0.3 monolith,
by collapsing retrieval, inference, and an implicit always-local assumption into
one place that bypasses routing, is incompatible with that intent.

## Decision

### 1. Complete the layer migration

The v0.3 monolith is decomposed so that **all** real local inference flows
through the execution layer (`LocalExecutor`). `ModelManager` is demoted from a
god-object to a model/resource provider (loaded model handle + RAGpack store). No
UI screen may invoke inference outside the coordinator.

### 2. Unify on `HybridExecutionCoordinator` as the canonical coordinator

`HybridExecutionCoordinator` becomes the single canonical coordinator. It is
already wired to `@main`; it already delegates to the canonical `Router` and
`PolicyEngine`; it owns `HumanOverrideMode` (ADR-0000 #4, Human Override
Authority); and it supports both `local` and `cloud` routes, which v0.4 requires.

### 3. Retire dead duplicates

- Delete `HybridRouter` and `HybridPolicyEngine` (self-declared unused).
- Delete the Preview-only non-Hybrid `ExecutionCoordinator` **after** its
  `executeLocal → ModelManager` wiring has been absorbed into the real
  `LocalExecutor` — it is presently the only correct reference wiring to real RAG,
  so deletion order is load-bearing.
- Canonical decision components are: `HybridExecutionCoordinator`, `Router`,
  `PolicyEngine`, `ConstraintRuntime`.

### 4. Enforce Privacy Step 4.5 (mandatory, non-bypassable)

On `privacy_level == local` (and AUTO resolving to local with
`fallbackAllowed == false`), the execution path MUST hard-block all network
egress. Enforcement MUST be non-bypassable and logged with `trace_id`. A routing
decision alone does not satisfy this requirement.

### 5. Cloud route remains in scope for v0.4

The `remoteAgent` route (`AgentExecutor` → `noema-agent`) is a first-class v0.4
route, selected only by deterministic Router policy or explicit
`HumanOverrideMode`. The default remains offline/local. The agent endpoint MUST
be a user-visible opt-in, and the routing decision and execution metadata MUST be
inspectable (ADR-0007 invariant).

## Rationale

### Why complete the migration rather than rewrite

The decision/routing core (`Router`, `PolicyEngine`, `ConstraintRuntime`, tracing)
was verified to be deterministic, side-effect-free, and ADR-0000/0005 compliant.
The defect is not in the architecture but in an unfinished execution-layer
migration. Discarding the compliant core would re-incur its cost for no benefit.

### Why `HybridExecutionCoordinator` and not the non-Hybrid coordinator

Audit initially suggested the non-Hybrid components were canonical. Code
verification refined this: `HybridRouter`/`HybridPolicyEngine` are dead, so
`Router`/`PolicyEngine` are canonical — but the **coordinator** is the inverse.
`HybridExecutionCoordinator` is the one wired to `@main`, it consumes the
canonical `Router`/`PolicyEngine`, it implements human override, and it carries
the cloud route. Because Decision 5 keeps the cloud route in scope, the Hybrid
coordinator is the correct canonical choice.

### Why enforce, not merely decide, privacy

ADR-0000 #7 holds that transparency and guarantees outrank optimization. A
privacy guarantee that is decided but unenforced is nominal, not real. The
"private, offline" identity (ADR-0001, ADR-0007) requires the local-only path to
be provably network-silent, verifiable in CI.

## Consequences

### Positive
- A single execution path; routing governs every query.
- The privacy guarantee becomes real and testable, not nominal.
- Dead code removed; the abandoned mid-migration state is resolved.
- v0.3 UI/UX preserved while gaining v0.4 routing.

### Trade-offs
- R1 (monolith decomposition) is non-trivial and must be done with discipline to
  avoid re-merging retrieval and inference concerns.
- Cloud route in scope means `noema-agent` integration and the `localhost:8080`
  default must be hardened into a deliberate, user-visible opt-in.

### Mitigations
- Strict step ordering (decompose before delete; see implementation plan).
- A UAT suite, run in CI, that locks the invariants: local route returns a real
  answer, local-only produces zero egress, routing is deterministic, no silent
  fallback, cloud is opt-in and inspectable.

## Compliance

This ADR **completes**, and does not amend, ADR-0000, ADR-0004, ADR-0005, and
ADR-0006. It operationalizes ADR-0007 ("NoesisNoema as a reusable private RAG
runtime") by first making the internal execution path contract-clean. Any
fallback between routes must remain explicit, Router-declared, and logged — never
silent (ADR-0000, Anti-pattern #2).
