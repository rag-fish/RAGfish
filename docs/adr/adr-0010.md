## Context

NoesisNoema for macOS currently renders only `MinimalClientView` â€” the EPIC1 vertical-slice
debug screen. `Shared/NoesisNoemaApp.swift` (the macOS `@main`) calls
`MinimalClientView(executionCoordinator:)` directly. There is no chat transcript view, no
tab/section navigation, no history browser, no settings panel â€” features that exist and
work on the iOS app since Phase 1 (PR #87).

The codebase contains a parallel macOS UI from the v0.22 era: `Shared/ContentView.swift`
(539 lines, gated by `#if os(macOS)`, imports AppKit). It implements a rich feature set:
embedding-model picker, LLM-model picker with autotune, LLM-preset picker, runtime-mode
picker (recommended/override + reset), Q&A history with selection, offline-mode display,
RAGpack manager toggle, autotune-status warnings, and a question/answer area.

However, `ContentView` predates the v0.4 layer migration (ADRs 0005â€“0009). It calls
`ModelManager.shared.generateAsyncAnswer()` **directly**, bypassing
`HybridExecutionCoordinator`, the Router/PolicyEngine, ADR-0008 R3 privacy enforcement, the
ADR-0009 session-memory path, and `ExecutionResult`/`NoemaResponse` citation plumbing. As
written it cannot honour the v0.4 architecture.

Three reconstruction options were considered:

- **(A) Wire `ContentView` straight to `@main`.** Cheapest, restores the v0.22 feature set
  immediately, but inherits v0.22 execution: no coordinator, no privacy enforcement, no
  multi-turn, no citations. The shipping app would be functionally inconsistent with iOS.
- **(B) Reuse the iOS `TabRootView` on macOS.** Lowest design cost, shares views, gives
  Mac users a phone-style bottom tab bar â€” visibly out of place on macOS and rejected by
  HIG conventions. Also entangles `Color(.systemBackground)` (iOS-only API) across both
  platforms.
- **(C) Build a native macOS shell with `NavigationSplitView`, reusing Shared
  coordinator/executor/session-memory infrastructure.** Highest design cost, but is the
  only option where macOS gets a Mac-shaped UX **and** the same v0.4 execution guarantees
  as iOS.

Option C is chosen because the goal is not to "make macOS launch" but to make macOS a
first-class peer of iOS under the v0.4 architecture. A Mac UI that ships without the same
privacy/routing/memory guarantees would dilute the product's central claim.

## Decision

### 1. macOS gets its own view layer; the runtime is shared

A new directory â€” provisionally `Apps/macOS/NoesisNoemaDesktop/Views/` â€” holds macOS
SwiftUI views. The execution path (`HybridExecutionCoordinator`, `LocalExecutor`,
`SessionMemory`, `NoemaRequest`, citations) is reused unchanged from `Shared/`. The
view layer differs by platform; the engine does not.

### 2. The macOS shell is a NavigationSplitView

`DesktopRootView` uses `NavigationSplitView` with:
- **Sidebar (leading column):** navigation between Chat, History, and Settings sections.
- **Detail (trailing column):** the section's content (chat transcript, history list +
  detail, settings form).

The iOS bottom-tab pattern (`TabRootView` + `TabBarView`) is NOT used on macOS.
Window sizing, toolbar, and standard Mac behaviours (resizable, multi-window if useful
later) follow macOS conventions.

### 3. macOS feature set = iOS feature set + the macOS-specific v0.22 features

The Chat section MUST support, at a minimum, what iOS supports today (PRs #87â€“#91):
- Conversation transcript (question bubble + answer, inline, auto-scroll)
- Coordinator-routed inference (`HybridExecutionCoordinator`)
- Session memory (ADR-0009 â€” 3 turns / 45-min window, identical caps as iOS)
- Citations propagated through `ExecutionResult`/`NoemaResponse`
- Llama 3.2 3B default with the Model/Preset header indicator

In addition, the following v0.22 macOS-only features MUST be preserved (sourced from
`Shared/ContentView.swift`):
- **Embedding model picker** (`ModelManager.availableEmbeddingModels`)
- **LLM model picker with autotune** (`autotuneCurrentModelAsync`, the `autotuneWarning`
  surface, and the `recommendedReady` indicator)
- **LLM preset picker** (`availableLLMPresets`)
- **Runtime mode picker** with Reset to recommended (`LLMRuntimeMode` â€” `.auto` /
  override; `setLLMRuntimeMode` / `resetToRecommended`)
- **Offline-fully-local indicator** (`appSettings.offline && ModelManager.isFullyLocal()`)
- **RAGpack manager access**

The History section reuses the iOS Q&A model (`documentManager.qaHistory`, `QAPair`,
citations, feedback đź‘Ť/đź‘Ž) but presents it in a macOS-natural way (list + detail in the
detail pane).

The Settings section consolidates iOS settings (DeepSearch toggle, Clear History) and the
macOS-specific pickers above. The header/in-place Model picker style of iOS
(`MobileHomeView.modelSelectorSection`) is replaced by Settings-resident pickers on macOS,
because a Mac window has room for them as first-class controls.

### 4. All inference goes through the coordinator

The macOS Chat view MUST call `executionCoordinator.execute(request: NoemaRequest(...))`,
populating `history` via `SessionMemory.history(from: documentManager.qaHistory)`. It
MUST NOT call `ModelManager.shared.generateAsyncAnswer` directly. This is the central
correction over `Shared/ContentView.swift`: same UI affordances, v0.4 execution
underneath.

### 5. macOS-iOS view code is duplicated, not shared

The iOS views in `Apps/iOS/NoesisNoemaMobile/Views/` are NOT reused on macOS. Two reasons:
- They use iOS-only APIs (`Color(.systemBackground)` in 8 sites) and an iOS-shaped layout
  (bottom tab bar, full-bleed sections) that would force `#if os(...)` everywhere.
- Cross-platform SwiftUI views ossify around the lowest common denominator. Keeping the
  view layers separate lets each platform express idiomatic UX without compromise.

The view layer therefore duplicates structure; the shared layer (everything below the
view) does not. This is an explicit, accepted trade-off.

### 6. The legacy ContentView is retired, not deleted (this PR)

`Shared/ContentView.swift` is removed from the app's render path (no `@main` wiring) but
the file is left in the repository for one release as a reference for v0.22 feature
behaviour during macOS reconstruction. A follow-up cleanup PR may delete it once the new
macOS UI has shipped and been UAT'd.

`MinimalClientView` is also retained for macOS, behind a `#if DEBUG` debug entry (mirroring
the iOS arrangement in PR #87) â€” it remains a useful vertical-slice debug screen.

## Rationale

- **Architectural consistency over UI reuse.** The product claim ("private, offline,
  audited routing, no hidden memory") only holds if every shipping surface enforces it.
  Letting macOS run on the v0.22 direct-`ModelManager` path would silently break that
  claim on one of two shipping platforms.
- **HIG appropriateness.** Mac users expect sidebars, resizable windows, and standard
  navigation; phone-style tab bars on macOS read as a port and erode trust in a
  professional tool. NavigationSplitView is the SwiftUI-native answer.
- **Feature non-regression.** v0.22 macOS already shipped richer controls (embedding/LLM
  pickers, autotune, runtime mode). Cutting these to "match iOS" would be a regression;
  preserving them is a hard requirement, not a stretch goal.
- **Why duplicate the view layer.** A single cross-platform view layer either compromises
  one platform's UX or is laced with `#if os(...)`. Sharing the runtime (the part where
  bugs cost the most) and forking the view layer (where idiomatic UX matters most) is the
  standard trade-off and matches what iOS already does in Apple's own apps.

## Consequences

### Positive
- macOS reaches feature parity with iOS plus retains the v0.22 advanced controls.
- All inference on macOS routes through the audited v0.4 execution path
  (coordinator / privacy / session memory / citations).
- The codebase has one runtime path, two idiomatic view layers â€” easier to reason about
  than a single platform-multiplexed view layer.

### Trade-offs
- View code is duplicated between iOS and macOS; bug-fixes in one section may need to be
  applied to the other (the runtime is shared, so most fixes happen below the view).
- Implementation cost is the largest of the three options.
- macOS Mac Catalyst is **not** chosen â€” Catalyst would have been a fourth option but
  produces a worse UX than native AppKit/SwiftUI and is out of scope.

### Mitigations
- The Chat / History / Settings split is small and well-bounded â€” each macOS section maps
  cleanly to an iOS counterpart, so structural divergence is limited.
- Where view-layer logic is genuinely identical between platforms (e.g. transcript
  rendering of a `[QAPair]`), small leaf views may be lifted to `Shared/` later; this is a
  refactor opportunity, not a requirement for the initial restoration.

## Compliance

Subordinate to ADR-0000 (Human Sovereignty â€” macOS must enforce the same guarantees as
iOS). Implements ADR-0005 (client-side routing â€” macOS now routes through the same
Router/PolicyEngine), ADR-0008 (layer migration completion â€” macOS Chat goes through
`HybridExecutionCoordinator`, not `ModelManager` directly), and ADR-0009 (session memory â€”
macOS Chat populates `NoemaRequest.history` from the visible transcript with the same
3-turn / 45-min caps). Does not amend ADR-0006 (Contract Lock) â€” no contract change is
introduced; the macOS UI consumes existing types.
